# Academic Year and Section Migration

## Purpose

This change adds Academic Year and Section support to the existing University ERP without changing existing API URLs, existing request formats, existing response formats, semester generation, or master-data ownership.

Academic Year is a teaching-session layer. Section is a student/faculty allocation layer inside a teaching session. Program, Semester, Subject, and Specialization remain master data.

## Existing Hierarchy

```text
University
`-- School
    `-- Program
        |-- Semester
        |   `-- Subject
        |       |-- Common Subject
        |       `-- Specialization Subject
        `-- Specialization
```

Existing user academic mapping:

```text
User
|-- schoolId
|-- academicAssignments[]
|   |-- programId
|   |-- semesterId
|   |-- specializationId, optional
|   |-- assignedSubjects[], faculty/coordinator only
|   |-- isCoordinator
|   |-- isPrimary
|   `-- status
`-- currentSemester
```

## New Hierarchy

```text
University
`-- School
    |-- AcademicYear
    |   |-- 2025-26
    |   `-- 2026-27
    `-- Program
        |-- Semester
        |   |-- Section
        |   |   |-- Students
        |   |   `-- Faculty Allocation
        |   `-- Subject
        |       |-- Common Subject
        |       `-- Specialization Subject
        `-- Specialization
```

Important ownership rules:

- Academic Year belongs to School.
- Program belongs to School.
- Semester belongs to Program.
- Subject belongs to Program, Semester, and optional Specialization.
- Section belongs to Academic Year, Program, Semester, and optional Specialization.
- Academic Year does not own Program.
- Academic Year does not own Semester.
- Academic Year does not own Subject.
- Section does not own Subject.
- Section consumes Subjects by matching Program + Semester + Specialization.

## ER Diagram

```text
School 1 ------------------ * AcademicYear
School 1 ------------------ * Program
Program 1 ----------------- * Semester
Program 1 ----------------- * Specialization
Program 1 ----------------- * Subject
Semester 1 ---------------- * Subject
Specialization 0..1 ------- * Subject

AcademicYear 1 ------------ * Section
Program 1 ----------------- * Section
Semester 1 ---------------- * Section
Specialization 0..1 ------- * Section

User 1 -------------------- * academicAssignments
User 1 -------------------- * teachingAssignments
academicAssignments * ----- 0..1 AcademicYear
academicAssignments * ----- 0..1 Section
teachingAssignments * ----- 0..1 AcademicYear
teachingAssignments * ----- 0..1 Section
teachingAssignments * ----- 1 Subject
```

## Added Model: AcademicYear

File: `backend/src/models/academicYear.model.js`

Collection model name: `AcademicYear`

Fields:

- `name`: string, required, trimmed. Example: `2025-26`.
- `startDate`: date, required.
- `endDate`: date, required.
- `status`: string enum, default `active`. Allowed values: `active`, `inactive`, `archived`.
- `schoolId`: ObjectId ref `School`, required.
- `isCurrent`: boolean, default `false`.
- `createdBy`: ObjectId ref `User`, default `null`.
- `updatedBy`: ObjectId ref `User`, default `null`.
- `createdAt`: generated by timestamps.
- `updatedAt`: generated by timestamps.

Indexes:

- `{ schoolId: 1, name: 1 }`, unique.
- `{ schoolId: 1, isCurrent: 1 }`, unique partial index where `isCurrent: true`.
- `{ schoolId: 1, status: 1 }`.
- `{ startDate: 1, endDate: 1 }`.

Business rules:

- One school can have many academic years.
- One school can have only one current academic year.
- Marking one academic year current automatically unsets other current years in the same school.
- Academic year creation does not create programs.
- Academic year creation does not create semesters.
- Academic year creation does not create subjects.

## Added Model: Section

File: `backend/src/models/section.model.js`

Collection model name: `Section`

Fields:

- `name`: string, required, trimmed. Example: `A`, `B`, `C`.
- `academicYearId`: ObjectId ref `AcademicYear`, required.
- `programId`: ObjectId ref `Program`, required.
- `semesterId`: ObjectId ref `Semester`, required.
- `specializationId`: ObjectId ref `Specialization`, optional, default `null`.
- `capacity`: number, default `0`, min `0`.
- `status`: string enum, default `active`. Allowed values: `active`, `inactive`, `archived`.
- `createdBy`: ObjectId ref `User`, default `null`.
- `updatedBy`: ObjectId ref `User`, default `null`.
- `createdAt`: generated by timestamps.
- `updatedAt`: generated by timestamps.

Indexes:

- `{ academicYearId: 1, programId: 1, semesterId: 1, specializationId: 1, name: 1 }`, unique.
- `{ academicYearId: 1, status: 1 }`.
- `{ programId: 1, semesterId: 1 }`.
- `{ specializationId: 1 }`.

Business rules:

- One semester can have many sections.
- Sections are session-specific because they belong to an academic year.
- Core sections use `specializationId: null`.
- Specialization sections use the matching `specializationId`.
- Section program must belong to the academic year's school.
- Section semester must belong to the section program.
- Section specialization, when present, must belong to the section program.

## Academic Year API Behavior

List:

- Supports existing REST list behavior.
- Search fields: `name`.
- Filters: `schoolId`, `status`, `isCurrent`.
- Sort fields: `name`, `startDate`, `endDate`, `status`, `createdAt`, `updatedAt`.
- Populates `schoolId`, `createdBy`, `updatedBy`.

Create:

- Roles allowed: `superAdmin`, `schoolAdmin`.
- Required body fields: `name`, `startDate`, `endDate`, `schoolId`.
- Optional body fields: `status`, `isCurrent`.
- Uses transaction.
- If `isCurrent` is true, unsets current academic year for same school.
- Writes `ACADEMIC_YEAR_CREATED`.

Update:

- Roles allowed: `superAdmin`, `schoolAdmin`.
- Same validation as create.
- Uses transaction.
- If `isCurrent` is true, unsets current academic year for same school except itself.
- Writes `ACADEMIC_YEAR_UPDATED`.

Delete:

- Roles allowed: `superAdmin`, `schoolAdmin`.
- Soft-deactivates by setting `status: inactive` and `isCurrent: false`.
- Writes `ACADEMIC_YEAR_DELETED`.

## Rollout Strategy

Recommended order:

1. Deploy backend code.
2. Let MongoDB build new indexes.
3. Create one academic year for each school.
4. Mark one academic year as current per school.
5. Create sections for each active program + semester + specialization combination that needs batches.
6. Gradually update student assignments with academic year and section.
7. Gradually use section-scoped faculty/coordinator assignment.
8. Enable frontend selectors for academic year and section when the admin UI is ready.

No data backfill is required before deployment.

## Potential Risks

- MongoDB unique indexes can fail to build if duplicate academic year or section data is inserted manually before deployment.
- Existing records do not automatically receive academic year or section values.
- Section delete is currently soft-delete for safety; hard delete would need cleanup rules for user assignments and teaching assignments.
- Frontend selectors for academic year and section must be added before admins can fully manage the new fields from UI.
- Mongoose transaction support requires MongoDB replica set or compatible transaction-capable deployment.

## Final Compatibility Statement

This migration is additive. Existing frontend integration remains valid. Existing business logic remains available. Existing users, programs, semesters, subjects, notices, notifications, dashboard widgets, and assignment flows continue to work without requiring immediate data migration.
